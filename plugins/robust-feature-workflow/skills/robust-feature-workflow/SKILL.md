# Robust Feature Workflow

하나의 기능을 **견고하게** 만들기 위한 단계별 워크플로우. 핵심 철학은 두 가지다.

1. **안에서 바깥으로 (inside-out).** 도메인 → 애플리케이션 → 어댑터 순. 의존성 방향과 작업
   순서를 일치시키면, 뒤 작업이 앞 작업의 검증된 결과 위에서 안정적으로 쌓인다.
2. **게이트로 끊는다.** 각 Phase 끝에 "여기까지는 깨지지 않는다"는 보장(끝 조건)을 만들고
   나서 다음으로 넘어간다. 견고함은 이 보장의 누적이다.

이 워크플로우는 며칠~2주가 걸릴 수 있다. 한 세션에 다 끝내려 하지 말고, **진행 파일**로 상태를
남기며 게이트 단위로 전진한다.

## 언제 쓰고, 언제 안 쓰는가

- **쓴다:** 실패 비용이 큰 새 기능을 처음부터 제대로 설계해 만들 때. 불변식·상태전이·외부
  시스템 연동이 얽힌 도메인(주문/결제/예약/정산 등).
- **안 쓴다:** 버그픽스, 리네임, 오타, 단일 함수 리팩토링, 1~2시간짜리 잡일. 이런 건 그냥
  바로 한다. 무거운 게이트를 강요하지 말 것.

판단이 애매하면 사용자에게 "이건 가볍게 갈까요, 워크플로우대로 탄탄하게 갈까요?"를 물어라.

## 시작: 진행 파일을 만든다

기능 이름을 정하고 리포 적절한 위치(예: `docs/features/` 또는 프로젝트 컨벤션)에
`FEATURE-<기능명>.md`를 만든다. `assets/feature-progress-template.md`를 복사해 채운다.

이 파일은 **세션 간 상태의 단일 출처**다. 매 Phase 끝 게이트를 통과하면 체크하고, 주요 결정을
한 줄씩 적는다. 새 세션에서 작업을 이어받을 때 가장 먼저 이 파일을 읽어 현재 Phase를 파악한다.

진행 파일이 이미 있으면: 읽고, 마지막으로 통과한 게이트의 **다음 단계**부터 시작한다. 앞
단계를 다시 하지 않는다.

## 작업 원칙 (4가지)

각 Phase 내내 지킨다.

- **테스트가 설계 압력이다.** "이걸 어떻게 테스트하지?"에 답이 안 나오면 설계가 틀린 신호다.
  포트가 빠졌거나 책임이 잘못 놓였다. 억지로 짜지 말고 설계를 고쳐라.
- **한 번에 한 가지 변경.** TDD 사이클 안에서도, PR 단위에서도. "리팩토링 + 기능 추가" 동시
  진행 금지 — 깨졌을 때 원인 추적이 폭발한다.
- **결정을 기록한다.** "왜 이렇게 설계했는가"는 6개월 뒤 까먹는다. ADR 한 페이지가 미래의
  본인과 동료를 구한다. 되돌리기 어려운 결정마다 진행 파일이나 ADR에 한 줄. **ADR·용어집·
  CLAUDE.md·README 같은 프로젝트 문서는 `domain-docs` 스킬로 작성한다** — 문서를 어느 계층에
  둘지(시스템 vs 도메인)와 템플릿은 그 스킬이 권위다. 이 워크플로우는 자체 산출물(진행 파일,
  Acceptance Criteria)만 직접 만든다.
- **게이트를 건너뛰지 않는다.** Phase 끝 조건이 안 잡혔으면 다음으로 넘어가지 않는다. 빨리
  가고 싶으면 이 워크플로우를 안 쓰면 된다. 쓰기로 했으면 게이트를 지킨다.

## Phase 0 — 기능 이해 (코딩 전)

코드 한 줄 짜기 전에 끝낸다. 여기가 흐릿하면 뒤가 다 무너진다.

1. **유스케이스를 한 문장으로.** "고객이 장바구니 상품을 선택해 배송지를 지정하고 주문을
   생성한다." 이 문장이 안 써지면 요구사항이 덜 정리된 것.
2. **흐름 정리 (Event Storming 약식).** 시간순으로 나열: 명령(Command), 도메인 이벤트,
   정책/규칙, 외부 시스템. 포스트잇 없이 진행 파일에 적어도 된다.
3. **경계 식별.** 이 기능의 Aggregate는? 다른 Aggregate는 ID로만 참조하는가? 트랜잭션
   경계는 어디까지인가?
4. **불변식(Invariants) 목록화.** 이게 나중에 그대로 테스트가 된다. (예: 주문 라인 1개 이상,
   총액 음수 불가, PLACED에서만 결제 가능, 배송 후 취소 불가, 동일 통화만 합산.)
5. **Acceptance Criteria 작성 (Given-When-Then).** 그대로 Phase 4 통합 테스트가 된다.
   양식은 `references/templates.md` 참고. `assets/acceptance-criteria-template.md`를 채워라.

**산출물:** ADR 한 장(`domain-docs` 스킬로 작성 — 배치 계층·템플릿은 그 스킬 참고),
Acceptance Criteria 문서(`assets/acceptance-criteria-template.md`).

> **게이트 0:** 유스케이스 한 문장 + 불변식 목록 + Given-When-Then 1개 이상이 진행 파일/문서로
> 존재한다. 트랜잭션 경계와 Aggregate가 명시됐다.

---

## Phase 1 — 도메인 모델링 (인사이드아웃 시작)

가장 안쪽 계층. 외부 세계의 어떤 것도 모른 채로 만든다.

1. **VO부터, 의존성 그래프의 잎(leaf)부터.** 예: `Money → Currency → ProductId → OrderId`.
   각각 TDD 사이클(Red → Green → Refactor)로.
2. **Entity → Aggregate Root.** `Order.place(...)` 정적 팩토리부터. 이 시점에 도메인 이벤트도
   등장한다.
3. **상태 전이를 테스트로 박는다.** 상태 머신을 표로 그리고 각 전이를 테스트 1개로 변환한다.
   허용/불허 모두 빠짐없이. 표 양식은 `references/templates.md` 참고.
4. **정책은 인터페이스로 분리.** `DiscountPolicy` 등은 도메인 내 인터페이스로 두고 기본 구현
   하나(`NoDiscountPolicy`)를 같이. 테스트에서 주입한다.

> **게이트 1:** 도메인 패키지에 인프라/프레임워크 의존성이 **0**이고, 모든 도메인 단위
> 테스트가 인메모리로 1초 안에 통과한다. 모든 불변식과 상태 전이가 테스트로 커버됐다.

---

## Phase 2 — Application Layer (유스케이스)

1. **인바운드 포트(UseCase 인터페이스) 정의.** 구현 없이 인터페이스만. 예:
   `PlaceOrderUseCase.placeOrder(PlaceOrderCommand)`.
2. **아웃바운드 포트 도출.** 유스케이스를 구현하며 필요해지는 외부 의존성마다 포트를 추가
   (`ProductCatalogPort`, `InventoryPort`, `OrderRepository`, `DomainEventPublisher`). **인터페이스만**
   먼저. 구현체는 Phase 3.
3. **Application Service 테스트 (모든 포트 Mock).** 성공 케이스 1개를 먼저 통과시키고, 그
   다음 실패/경계: 재고 부족→저장 안 됨, 결제 실패→재고 롤백, 잘못된 배송지→검증 실패.
4. **트랜잭션 경계 결정.** 보통 Application Service 메서드. 이 결정도 ADR/진행 파일에 한 줄.

> **게이트 2:** Mock 기반 Application 단위 테스트가 모두 통과. DB·HTTP 없이 비즈니스 시나리오
> 전체가 검증됨. 필요한 아웃바운드 포트가 인터페이스로 다 도출됐다.

---

## Phase 3 — Adapter (바깥쪽으로)

이제 실제 외부 세계와 붙인다. 어댑터별로, 각각 통합 테스트와 함께.

1. **Persistence Adapter.** JPA Entity와 도메인 모델 분리, Mapper, `OrderRepository` 구현.
   round-trip 테스트(저장→조회→동일 도메인 복원). 흔한 함정: 컬렉션 cascade/orphanRemoval,
   VO의 `@Embeddable`, 도메인 이벤트는 영속화 금지(transient).
2. **외부 시스템 Adapter.** HTTP 클라이언트(타임아웃/재시도/서킷브레이커), PG 연동(실패 처리).
   어댑터마다 계약 테스트(WireMock 등).
3. **Web Adapter.** DTO와 도메인 분리(Request/Response는 web 패키지에만), 입구에서 검증,
   도메인 예외 → HTTP 상태 매핑(`@ControllerAdvice` 등). UseCase는 Mock으로 계약만 검증.

스택별 구체 도구(@DataJpaTest, @WebMvcTest, WireMock 등)는 `references/spring-stack.md` 참고.
다른 스택이면 그 스택의 동등 도구로 같은 역할을 채운다.

> **게이트 3:** 각 어댑터별 통합 테스트 통과. 단 아직 전체 흐름은 안 돌려봤다.

---

## Phase 4 — End-to-End 검증

1. **실제 인프라 기반 통합 테스트.** Phase 0의 Acceptance Criteria가 그대로 테스트가 된다.
   실제 DB(Testcontainers 등) + 실제 컨텍스트, 외부 시스템만 WireMock으로 대체. 느리니 CI나
   별도 프로파일에서.
2. **동시성·장애 시나리오.** 같은 재고 동시 주문(낙관/비관 락), 외부 응답 지연(타임아웃+재시도),
   이벤트 발행 후 롤백(Transactional Outbox 고려).
3. **성능 측정 (필요 시).** P99 지연 측정, 병목 발견 시 인덱스/쿼리/캐시.

> **게이트 4:** Acceptance Criteria가 E2E 테스트로 통과한다. 기능이 살아 있다는 신호.

---

## Phase 5 — 비기능 요구사항 보강

- **관찰성:** 구조화 로깅(orderId/customerId를 MDC에), 메트릭(생성 카운트/실패율/지연),
  분산 트레이싱.
- **보안:** 입력 검증 재점검, 인증·인가(타인 주문 조회 차단), 민감정보 로깅 금지.
- **문서화:** OpenAPI 스펙, README 실행법/환경변수, 주요 결정 ADR.

> **게이트 5:** 로깅·메트릭·인가가 들어갔고, 외부에 노출되는 계약(API)이 문서화됐다.

---

## Phase 6 — 코드 리뷰 & 리팩토링

- **셀프 리뷰 체크리스트** (`references/templates.md`): 도메인에 프레임워크 의존성 없음,
  비즈니스 규칙 중복 없음, 테스트가 구현이 아닌 행위를 검증, 이름이 도메인 언어와 일치,
  죽은 코드/의미 없는 TODO 정리.
- **PR 쪼개기:** 1 PR = 1 관심사. 리뷰어가 30분 안에 읽을 크기.
- **마지막 리팩토링:** 테스트가 보장된 상태에서만. 이름·중복·추상화 수준 정리.

> **게이트 6 (완료):** 셀프 리뷰 체크리스트 전부 통과, PR이 관심사별로 쪼개져 올라감.

## 시간 분배 감각 (2주 기준)

| Phase | 비중 |  | Phase | 비중 |
|---|---|---|---|---|
| 0. 이해 | 10% |  | 4. E2E | 10% |
| 1. 도메인 | 25% |  | 5. 비기능 | 10% |
| 2. App | 20% |  | 6. 리뷰 | 5% |
| 3. Adapter | 20% |  | | |

처음엔 Phase 0·1에 더 쓰는 게 정석. 여기가 흔들리면 뒤에서 몇 배로 갚는다.

## 참고 자료

- `references/templates.md` — ADR/Given-When-Then 작성법, 상태전이 테스트 표 패턴, 포트
  인터페이스 스텁 예시, 셀프 리뷰 체크리스트.
- `references/spring-stack.md` — Phase별 Spring/JPA/Testcontainers/WireMock 구체 도구. 다른
  스택이면 동등 도구로 치환.
- `assets/` — 복사해서 채우는 워크플로우 산출물 템플릿:
  `acceptance-criteria-template.md`, `feature-progress-template.md`.
- **문서(ADR·용어집·CLAUDE.md·README)는 `domain-docs` 스킬** — 배치 계층 판단과 템플릿을
  그 스킬이 제공한다. 이 워크플로우에서 ADR을 쓸 땐 `domain-docs`를 사용한다.
