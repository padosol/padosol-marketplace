# Domain Docs — 문서 배치 & 작성 전략

도메인이 늘어나는 코드베이스에서 문서를 **어디에, 어느 단위로 두느냐**가 유지보수성을 가른다.
이 스킬은 그 판단과 작성을 돕는다.

## 핵심 원칙 — 문서는 "변경 주체"를 따라간다

문서를 어디 둘지의 판단 기준은 단 하나다.

> **"이 문서가 변경될 때, 어떤 코드가 같이 변경되는가?"**

같이 변경되는 코드와 같은 곳에 둬야 한다. 안 그러면 코드와 문서가 어긋난다(co-location 원칙).
어느 계층인지 헷갈리면 이 질문 하나로 환원하라.

## 배치 결정표

| 변경 주체 | 위치 | 예시 |
|---|---|---|
| 전체 시스템 | 루트 `docs/`, 루트 `CLAUDE.md` | 빌드/배포, 전체 아키텍처, 기술 스택 |
| 여러 도메인 공통 | 루트 `docs/` | 공유 코드 규약, 도메인 간 통신 규약 |
| 특정 도메인만 | `{도메인}/docs/`, `{도메인}/CLAUDE.md` | Order 상태 머신, Product 카탈로그 정책 |
| 특정 어댑터만 | 어댑터 폴더 `README` (필요시만) | PG 연동 결정, JPA 매핑 결정 |
| 도메인 간 계약 | 루트 `docs/contracts/` | 발행 이벤트, 공유 ID/VO |

**판단법(ADR 등 결정 문서):** "이 결정을 바꾸려면 다른 도메인 코드도 건드려야 하나?"
→ Yes면 시스템 레벨(`docs/adr/`), No면 도메인 레벨(`{도메인}/docs/adr/`).

## 권장 디렉토리 구조

```
project/
├── README.md                    # 시스템 진입점 (신규자 30분 파악)
├── CLAUDE.md                    # 루트 AI 컨텍스트 (공통 규약·아키텍처)
├── docs/
│   ├── architecture.md          # 전체 아키텍처 + 다이어그램
│   ├── glossary.md              # 시스템 전체 용어집
│   ├── conventions.md           # 코딩/테스트/네이밍 규약
│   ├── contracts/               # 도메인 간 계약
│   └── adr/                     # 시스템 레벨 ADR + README(인덱스)
│
└── {도메인}/                     # 예: order/, product/
    ├── README.md                # 도메인 진입점 (사람용, 서술적)
    ├── CLAUDE.md                # 도메인 AI 컨텍스트 (AI용, 규칙형)
    ├── docs/
    │   ├── domain-model.md      # Aggregate, 상태 머신, 불변식
    │   ├── glossary.md          # 도메인 용어
    │   └── adr/                 # 도메인 레벨 ADR + README(인덱스)
    └── src/...
```

멀티 모듈(Gradle 등)로 가도 **모듈 단위 = 문서 단위**를 유지한다. 루트 `docs/`는 그대로 두고,
각 모듈 안에도 `docs/`·`CLAUDE.md`를 둔다. 모듈을 추출·통합할 때 문서가 자연스럽게 따라간다.

## CLAUDE.md 계층 — 누적 읽기를 활용한다

Claude Code는 작업 디렉토리부터 루트까지 거슬러 올라가며 `CLAUDE.md`를 **누적해서** 읽는다.
이걸 활용해 컨텍스트를 계층화한다.

- `order/`에서 작업 → 루트 `CLAUDE.md` + `order/CLAUDE.md`만 들어온다.
- 다른 도메인 디테일을 매번 끌어들이지 않아 토큰 효율·일관성이 좋아진다.

루트엔 공통 규약/아키텍처, 도메인 폴더엔 그 도메인 특화 컨텍스트(핵심 개념, 상태 전이, 이벤트,
Aggregate 경계, 포트, 주의사항). 작성법·템플릿은 `references/claude-md.md` 참고.

## ADR — 두 층위로 관리

시스템 ADR(`docs/adr/`)과 도메인 ADR(`{도메인}/docs/adr/`)을 구분한다. 가볍게(한 페이지 이내)
유지하고, "결정"과 "왜"만 명확히. 폐기된 ADR은 **삭제하지 말고** 상태만 `Superseded`로. 과거
맥락이 미래 의사결정의 자료가 된다. 2층위 판단·인덱스·코드 참조는 `references/adr.md` 참고.

## 흔한 함정 (반드시 피한다)

- **모든 결정을 ADR로 남기려는 강박.** 일상 코드 결정까지 ADR로 쓰면 ADR 자체가 안 읽힌다.
  기준: **"이 결정을 6개월 뒤 누군가 의문을 가질 만한가?"** 그런 것만 ADR로.
- **CLAUDE.md를 너무 자세하게.** 모든 규칙을 한 파일에 박으면 200줄을 넘기고 AI가 컨텍스트
  절반을 여기 쓴다. **AI가 일관성 있게 작업할 최소 정보**만. 각 CLAUDE.md 100~200줄 이내 목표.
- **README와 CLAUDE.md 중복.** README는 사람용(서술적), CLAUDE.md는 AI용(명령형/규칙형).
  청자와 의도가 다르므로 일부 겹쳐도 괜찮지만, 같은 내용을 복붙하지는 말 것.

## 문서가 썩지 않게 (변경 관리)

가장 큰 적은 "코드는 변경됐는데 문서는 그대로". 장치:
- **PR 체크리스트**로 문서 동기화 강제(도메인 규칙 변경→ADR, 새 용어→glossary 등).
- **ADR "결과" 섹션에 코드/테스트 경로**를 적어 둔다 → 코드 검색 시 ADR이 같이 발견됨.
- **CLAUDE.md는 짧게** 유지(길면 안 읽히고, 안 읽히면 갱신도 안 됨).
- **분기별 30분 점검**으로 폐기된 결정·낡은 컨텍스트 정리.

## 언제 쓰고, 언제 안 쓰는가

- **쓴다:** 결정을 ADR로 기록할 때, 문서를 어느 계층에 둘지 정할 때, 도메인별 CLAUDE.md·README·
  glossary·contracts를 만들 때, 문서 구조를 처음 세우거나 도메인 추가로 재정비할 때.
- **안 쓴다:** 자유형 마크다운 초안(→ `doc-draft`), 기존 CLAUDE.md 품질 감사·개선
  (→ `claude-md-improver`). 이 스킬은 **배치 전략 + 문서 타입별 템플릿 생성**이 본분.

## 작업 절차

1. 무엇을 기록/작성할지 파악하고, **배치 결정표**로 위치(루트 vs 도메인 vs 어댑터 vs contracts)를 정한다.
2. 위치가 비어 있으면 권장 구조대로 디렉토리를 만든다(이미 있으면 컨벤션을 따른다).
3. 해당 문서 타입의 `assets/` 템플릿을 복사해 채운다. 작성 디테일은 `references/` 참고.
4. ADR을 추가했으면 그 계층의 `adr/README.md`(인덱스)도 갱신한다.
5. 새 용어가 생겼으면 해당 계층 `glossary.md`에 반영한다.

## 참고 자료

- `references/adr.md` — ADR 2층위 판단, 인덱스, Superseded, 코드 참조, 가볍게 유지하는 법.
- `references/claude-md.md` — 루트/도메인 CLAUDE.md 작성법, 누적 읽기, 짧게 유지.
- `references/readme-glossary-contracts.md` — README(루트/도메인) 역할, glossary 작성, 도메인 간
  계약 배치, PR 체크리스트 등 변경 관리.
- `assets/` — 복사용 템플릿: `adr-template.md`, `adr-index-template.md`,
  `root-claude-md-template.md`, `domain-claude-md-template.md`, `domain-readme-template.md`,
  `glossary-template.md`.
