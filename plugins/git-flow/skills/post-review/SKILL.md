# post-review

코드 리뷰 보고서를 PR/MR 코멘트로 게시한다. 사용자가 머지를 결정하려면 리뷰 결과를 host 웹에서 볼 수 있어야 하는데, simplify-save 같은 스킬은 로컬 markdown 만 만들고 끝나기 때문에 그 결과를 PR/MR 로 옮기는 단계가 따로 필요해 만든 스킬.

## 언제 트리거

- `/post-review` (인자 있거나 없거나)
- "리뷰 결과 PR 에 올려" / "review 코멘트 달아" / "GitHub/GitLab 에 게시"
- `/simplify` 직후 사용자가 "PR 까지 올려" 의도 표현 → simplify-save 가 chain 호출

자동 트리거 금지: 사용자가 명시 의사 없으면 게시 안 함 (PR/MR 코멘트는 외부 가시 액션).

## 인자

```
/post-review [보고서경로] [PR번호]
```

- 둘 다 생략 가능. 결정 우선순위는 섹션 1 참조.
- 명시 예: `/post-review docs/simplify/2026-05-07-pad-7.md 84`

## 절차

### 0. 플랫폼 탐지 → 가이드 선택

어떤 git 플랫폼인지 **한 번** 결정한다 (결정 순서: `GF_HOST` 환경변수 > `.orch/settings.json` 의 `git_host` > `gh`/`glab` 인증 상태):

```bash
GF_HOST="${GF_HOST:-}"
if [ -z "$GF_HOST" ]; then
  d="$PWD"; orch=""
  while [ "$d" != "/" ]; do
    [ -f "$d/.orch/settings.json" ] && { orch="$d/.orch/settings.json"; break; }
    d="$(dirname "$d")"
  done
  if [ -n "$orch" ] && command -v jq >/dev/null 2>&1; then
    case "$(jq -r '.git_host // empty' "$orch" 2>/dev/null)" in
      github|gitlab) GF_HOST="$(jq -r '.git_host' "$orch")" ;;
    esac
  fi
fi
if [ -z "$GF_HOST" ]; then
  gh_ok=0; glab_ok=0
  command -v gh   >/dev/null 2>&1 && gh   auth status >/dev/null 2>&1 && gh_ok=1
  command -v glab >/dev/null 2>&1 && glab auth status >/dev/null 2>&1 && glab_ok=1
  if   [ "$gh_ok" = 1 ] && [ "$glab_ok" = 0 ]; then GF_HOST=github
  elif [ "$gh_ok" = 0 ] && [ "$glab_ok" = 1 ]; then GF_HOST=gitlab
  elif [ "$gh_ok" = 1 ] && [ "$glab_ok" = 1 ]; then
    echo "ERROR: gh / glab 양쪽 인증 — GF_HOST=github|gitlab 로 명시하거나 .orch/settings.json 의 git_host 설정" >&2; exit 2
  else
    echo "ERROR: gh / glab 모두 미인증 — 'gh auth login' 또는 'glab auth login' 후 재시도" >&2; exit 2
  fi
fi
case "$GF_HOST" in github|gitlab) ;; *) echo "ERROR: GF_HOST='$GF_HOST' 잘못된 값 (github|gitlab)" >&2; exit 2 ;; esac
echo "GF_HOST=$GF_HOST"
```

결정된 플랫폼에 따라 **이후 모든 호스트 명령은 이 스킬 디렉토리의 해당 가이드를 그대로 따른다**:
- `github` → `GITHUB_GUIDE.md`
- `gitlab` → `GITLAB_GUIDE.md`

먼저 해당 가이드를 Read 로 열어 둔다. 본문의 **`<가이드: 섹션명>`** 은 선택된 가이드의 동명 섹션 명령. **SKILL 본문에는 플랫폼 분기를 두지 않는다 — 분기는 가이드 파일에만.**

이하 본문에서 **`PR`** 은 GitHub PR / GitLab MR 양쪽을 의미.

### 1. 대상 결정

#### PR 번호

1. 인자에 PR 번호 있으면 그대로
2. 없으면 현재 브랜치의 open PR/MR 을 `<가이드: 현재 브랜치의 열린 PR 번호>` 로 검색
3. 검색 실패 → 사용자에게 묻고 중단

#### 보고서 경로

1. 인자에 경로 있으면 그대로 (존재 검증 필수)
2. 없으면 `docs/simplify/` (또는 사용자가 simplify-save 에 넘긴 출력 디렉토리) 의 가장 최근 `.md` 파일:
   ```bash
   ls -t docs/simplify/*.md 2>/dev/null | head -1
   ```
3. 못 찾으면 사용자에게 simplify-save 먼저 실행하도록 안내 후 중단

### 2. 사전 검증

`<가이드: PR 상태 조회>` 로 `state` 를 얻는다 (정규화: `OPEN`/`MERGED`/`CLOSED`).

- `state` 가 `OPEN` 이 아니면 자동 진행 금지. **`AskUserQuestion` 도구로 TUI 선택지 제시** — header `Non-OPEN PR`, options: `그래도 게시 (Recommended)` / `중단` / `다른 PR 지정`. 사용자가 "다른 PR 지정" 을 고르면 새 PR 번호를 자유 입력으로 받아 1단계부터 다시. plain text 질문 금지.
- 보고서 파일 size 가 0 이면 중단

### 3. 코멘트 본문 조립

마커 형식: `<!-- padosol-review:<source> -->` 첫 줄에 삽입. `<source>` 는 보고서가 어느 스킬 출신인지 (`simplify`, `audit`, ...). 디폴트 `simplify`.

본문 골격:

```markdown
<!-- padosol-review:simplify -->
### Code Review · Padosol simplify

> **Source:** `docs/simplify/<file>.md`
> **Posted:** <YYYY-MM-DD HH:MM UTC>
> [**Updated:** <YYYY-MM-DD HH:MM UTC> — 갱신 호출일 때만]

[**Issues:** <N> reuse · <M> quality · <K> efficiency · **Build:** <SUCCESS/FAILED>]

---

<보고서 markdown 본문 — 또는 장문 시 collapsible 감싸기 (아래 규칙 참고)>

---

<sub>🤖 Generated by Padosol</sub>
```

규칙:
- 마커는 **반드시 첫 줄**. idempotent update 의 anchor.
- 메타 banner (`> Source/Posted/Updated`) 는 blockquote 1개로 묶어 시각적 분리. Updated 라인은 4단계의 EXISTING 분기에서 update 일 때만 추가
- **Issues 한 줄 요약** 은 보고서 첫 줄/메타에 `**Issues:**` 가 있으면 추출해 banner 다음에 한 줄로 노출 (없으면 생략). 독자가 본문을 펼치기 전 신호로 활용
- Claude / Anthropic 브랜딩 (`🤖 Generated with Claude Code`, `Co-Authored-By: Claude ...`) 절대 금지. 푸터는 `🤖 Generated by Padosol` 만
- **장문 본문 처리** (보고서 본문 length > 8000 chars):
  ```markdown
  <details>
  <summary><strong>전체 리뷰 펼치기</strong> (<len> chars)</summary>

  <보고서 본문 그대로>

  </details>
  ```
  코멘트가 길어지면 스크롤 피로가 큼. 8000 이하면 그대로 노출 (collapsible 불필요)
- 본문이 host 코멘트 한도를 넘으면 (GitHub 65536 / GitLab 1_000_000) 헤더 + Issues 요약 + "전체는 `<file>` 참조" 로 축약 (가드)

### 4. 기존 코멘트 검색 → update or create

`<가이드: 마커로 기존 코멘트 찾기>` 로 마커 prefix `<!-- padosol-review:simplify -->` 로 기존 코멘트/노트 id 를 검색한다 (없으면 빈 문자열 → `EXISTING`).

분기:

#### 4-A. EXISTING 있으면 (update)

PATCH/PUT 직전에 본문 banner blockquote 에 `**Updated:** <YYYY-MM-DD HH:MM UTC>` 라인을 추가해 재조립한 뒤, `<가이드: 코멘트 갱신>` 으로 `EXISTING` id 의 코멘트를 갱신 (입력: PR 번호, 코멘트 id, 본문 파일). 독자가 PR 타임라인 알림 없이도 코멘트가 갱신됐음을 인지할 수 있도록.

#### 4-B. EXISTING 없으면 (create)

`<가이드: 코멘트 생성>` 으로 새 코멘트/노트를 만든다 (입력: PR 번호, 본문 파일 → 출력: 코멘트 id).

> 본문은 항상 **파일 경유** 로 전달한다 (`/tmp/post-review-body.md` 등). 따옴표/백틱 escape 사고 방지.

### 5. 결과 보고

생성/갱신된 코멘트 URL 캡처해 사용자에게 안내:

```
PR #<번호> 에 리뷰 코멘트 <create|update>:
  <PR URL>#<comment-anchor>

본문 출처: docs/simplify/<file>.md
```

코멘트 anchor 형식은 `<가이드: 코멘트 앵커 형식>` 참조 (GitHub `#issuecomment-<id>`, GitLab `#note_<id>`).

## 멱등성 (idempotency)

같은 PR 에서 재실행 → 마커로 기존 코멘트 찾아 갱신. 신규 코멘트 누적 금지. 다른 source (예: `audit`) 는 다른 마커이므로 별도 코멘트.

마커 변경 시 기존 코멘트는 고아가 됨 — 그래서 마커는 정해진 형식 그대로 유지하고 source 만 갈아낄 것.

## 실패 모드 / 주의

- ❌ 플랫폼 분기를 SKILL 본문에 인라인 — 분기 산개
  ✅ §0 에서 1회 탐지 후 GITHUB_GUIDE.md / GITLAB_GUIDE.md 의 명령만 사용 (id 추출 / PATCH·PUT 차이는 가이드가 흡수)
- ❌ Claude 브랜딩 푸터 추가 — 프로젝트 컨벤션 위반
  ✅ `🤖 Generated by Padosol` 만
- ❌ MERGED 또는 CLOSED PR 에 무단 코멘트 — 의미 없거나 노이즈
  ✅ state 검증, 비-OPEN 이면 사용자 확인 후에만
- ❌ 마커 누락 — 다음 호출이 중복 코멘트 생성
  ✅ 첫 줄 마커 필수, 검증
- ❌ `gh pr comment --edit-last` 또는 `glab mr note --edit-last` — "본인의 마지막 코멘트" 만 갱신, 다른 코멘트가 사이에 끼면 엉뚱한 걸 덮어씀
  ✅ 마커 검색 + 명시적 ID 갱신 (`<가이드: 코멘트 갱신>`)
- ❌ 보고서 본문을 그대로 셸 인자에 인라인 — 따옴표/백틱 escape 사고
  ✅ 항상 파일 경유 (가이드의 생성/갱신 명령이 본문 파일을 받음)

## 사용 예

### 예시 1: simplify 직후 chain (GitHub)

```
사용자: /simplify
Claude: [simplify-save] → docs/simplify/2026-05-07-pr-84-review.md 생성
사용자: 이거 PR 에도 올려줘
Claude: [/post-review 호출, 인자 없음]
        → §0 플랫폼 탐지 (GF_HOST=github) → GITHUB_GUIDE.md 채택
        → 현재 브랜치 PR #84 자동 검색
        → docs/simplify 최신 파일 자동 선택
        → state=OPEN 검증
        → 마커 검색: 없음 → <가이드: 코멘트 생성>
        → URL 보고
```

### 예시 2: GitLab MR + 재실행 (update)

```
사용자: /post-review docs/simplify/2026-05-07-pad-7.md 12
Claude: → §0 탐지 (GF_HOST=gitlab) → GITLAB_GUIDE.md
        → 마커 검색: 있음 (note id=987654)
        → <가이드: 코멘트 갱신> (note 987654)
        → URL 보고 (같은 note id, 갱신됨)
```
